# Social Account Operating Model

Historical archive only.
This note is kept for continuity and should not be treated as active runtime authority or default read-path material.

## Intent

Keep social publishing lanes separated before more automation lands.

One publish pipeline may serve multiple accounts, but the **source**, **tone**,
and **approval meaning** must stay account-specific.

At the same time, the three accounts are **not three unrelated identities**.
They share one philosophical spine and one recurring set of questions.
What changes is the narrator seat, the proof layer, and the surface texture.

## Shared Essence

Common core across `97layer`, `woosunhokr`, and `woohwahae`:

- subtraction over noise
- calm over performance
- proof over inflated claim
- daily practice over abstract slogan
- one clear observation before a bigger statement

Legacy spine behind that shared core:

The important reading is not “three brands.”
It is closer to:

- `97layer` = the raw operator / builder / diarist seat
- `woosunhokr` = the refined personal craft seat
- `woohwahae` = the polished public shell

So the essence stays shared, because the authorial spine is shared.
Only the level of polish, the visible proof, and the narrative seat change.

Operationally:

- the accounts should feel related
- they should not sound copy-pasted
- the same underlying thought may reappear, but the angle and vocabulary should
  shift by account

## Hard Rules

1. Do not treat one good post as reusable across all accounts.
2. Default to **no automatic cross-posting** between these three accounts.
3. Each account needs its own source stack before automation becomes routine.
4. If a post is created for one account and reused elsewhere, that should be a
   conscious rewrite, not a token swap.
5. Shared essence is allowed; shared wording is not the default.

## Operational Consequence

When the active token changes, the draft posture must change with it.

The same Threads publishing route can stay canonical, but the draft generator
and approval intent should read the destination account first.

The current practical gradient is:

- `97layer` -> raw
- `woosunhokr` -> refined
- `woohwahae` -> polished

The routing model should also change:

- external links and outside texts land in `97layer` first
- founder still decides what deserves translation
- approvals should stay lightweight and eventually happen in Telegram
- `woosunhokr` and `woohwahae` should receive rewritten versions, not raw feed dumps
