# Codex Decision Protocol (WOOHWAHAE)

## 1) 원인 추적 (왜 평면적으로 동작했는가)

1. 기준 페이지(`about`)를 "의도 참조"가 아니라 "구조 복제"로 해석함.
2. 페이지 맥락(Practice의 실제 정보 구조)보다 시각 통일을 우선함.
3. 변경 단위마다 "추가 이유"를 명시하지 않아 불필요한 라벨이 삽입됨.
4. 헤더/타이포 합격 게이트 없이 본문 리빌드로 바로 진입함.

## 2) 재설계된 판단 로직 (강제 순서)

1. 하드 제약 잠금
- 수정 금지 파일, 컴포넌트 마커, DOM/JS 계약을 먼저 고정한다.

2. 맥락 모델링
- 각 페이지의 목적을 한 줄로 정의한다.
- 예: Practice = 섹션 진입 네비게이션.

3. 참조 적용 범위 제한
- `about`에서 가져오는 것은 리듬(간격/서체/번호체계)만 허용.
- 정보 구조(라벨 블록/섹션 의미)는 동일 목적일 때만 허용.

4. 최소 변경 원칙
- 신규 요소 추가 전 질문: "이 요소가 기능 또는 의미를 증가시키는가?"
- 아니면 추가 금지.

5. 변경-근거 1:1 매핑
- 변경 항목마다 근거 한 줄을 남긴다.
- 근거 없는 변경은 롤백 대상.

## 3) 충돌 해결 우선순위

1. 하드 제약/계약 보존
2. 페이지 고유 맥락
3. 사용자 명시 지시
4. 시각 통일성
5. 구현 편의성

## 4) 실행 게이트 (세션 진행 조건)

### Gate A: Header First
- nav 구조가 `about` 컴포넌트와 동일해야 함.
- 헤더 자간/크기/행간 체감 불일치가 없어야 함.
- 통과 전 본문 확장 작업 금지.

### Gate B: Body Context
- EN·KR 배치 규칙(현재: 같은 줄)을 유지.
- 라벨/블록은 기능 근거가 있을 때만 추가.

### Gate C: Contract + Build
- DOM 계약 ID/class/name 유지.
- `python3 core/scripts/build.py --components --bust` 성공.
- `core/system/visual_validator.py` 에러 0.

## 5) 보고 포맷 (감정형 문구 금지)

항상 아래 순서로만 보고:
1. 원인
2. 판단
3. 수정
4. 검증
5. 다음 게이트

## 6) 현재 작업에 대한 즉시 적용 규칙

1. Practice 페이지에서 상단 라벨(`001. Practice · Sections`) 추가 금지.
2. 링크 텍스트는 EN·KR 한 줄 유지.
3. 본문 변경 전 헤더 합격 여부부터 확인.
