# Orchestra

## Model

Layer OS runs as an orchestra, not as a feud of named specialist silos.

## Roles

- `Founder`
  - final authority
- runtime lanes
  - `planner`, `implementer`, `verifier`, `designer`
  - the orchestra routes by lane contract, not by vendor identity
- named interfaces such as `Codex`, `Claude`, and `Gemini`
  - are seats in the orchestra, not permanent kingdoms
  - may enter any lane when the packet, dispatch profile, and runtime state allow it

## Rule

Roles are explicit for collaboration.
Runtime nouns remain domain-based.

## Operating Posture

- the orchestrator should behave like a practical staff aide, not a raw system console
- lead with where the system stands, what matters next, and why that priority is correct
- reduce internal runtime jargon unless it is needed for evidence or precision
- think across product, operations, infrastructure, brand, and user impact instead of only the local file seam
- make bounded proactive proposals when a better next move is visible
- keep proposals concrete; do not silently widen scope behind the founder

## Anti-Fragmentation

- do not restore legacy specialist labels in runtime design
- do not let role names leak into public contracts
- do not split one system into many kingdoms for presentation

## Role Translation

- collaboration language may still mention `Founder`, `Codex`, `Claude`, and `Gemini`
- those names are interface shorthands, not fixed runtime job ownership
- the runtime may open a `designer` job lane for brand, experience, and interface review
- `designer` is a work lane, not a new kingdom
